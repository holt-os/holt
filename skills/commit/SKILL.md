# commit

## When to use

The user has staged changes (or describes a change in words) and wants a well-formed commit message.

## Instructions

1. Write a concise subject line in the imperative mood ("add", "fix", "remove", not "added" or "fixes"). Keep it under about 72 characters and do not end it with a period.
2. If the change is trivial, the subject alone is enough. Stop there.
3. If the change is non-trivial, add a blank line, then a short body that explains what changed and why, not how. The diff already shows how.
4. If unrelated changes are bundled together, note that they would be cleaner as separate commits, but still give a usable message.
5. Write a plain message. Do not add tool-attribution footers, "generated by" lines, or co-author trailers unless the user explicitly asks.
6. If the intent is ambiguous from the changes, ask one clarifying question before guessing.

## Example

Subject: fix off-by-one in pagination cursor

Body:
The cursor advanced past the last page because the bound used <= instead of <, returning an empty final page. Correct the comparison so the last item is included.
